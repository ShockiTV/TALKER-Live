## Context

`texts/backstory/unique.py` contains ~120 rich, hand-written backstory paragraphs keyed by tech_name (e.g., `esc_2_12_stalker_wolf` → full paragraph about Wolf). When the LLM first encounters a unique NPC, it calls `background(action="read")` and gets back null from `memory_store_v2`. It then has to invent a background from scratch using `background(action="write")`, wasting a tool-call round and producing content that may contradict established lore.

Rather than intercepting this at runtime in Python, we pre-generate a static Lua data file with GM-style backstories, personality traits, and cross-referenced connections. This data is seeded into `memory_store_v2` as a one-time migration on first game start.

## Goals / Non-Goals

**Goals:**
- Pre-seed structured backgrounds for all ~120 unique NPCs on first game start
- Backstories rewritten in GM briefing style: atmospheric, dramatic, emphasizing personality hooks
- Traits extracted as personality adjectives (3-6 per NPC)
- Connections cross-referencing other unique NPCs, parsed from text mentions + STALKER universe knowledge
- Seeded backgrounds are first-class — identical in shape to LLM-written backgrounds, no special markers
- After seeding, standard background(read/write/update) flow handles everything

**Non-Goals:**
- Any Python-side changes (no intercepting `_handle_background`, no `resolve_backstory` changes)
- Runtime seeding or lazy seeding on first encounter
- Seeding backgrounds for generic NPCs (only unique NPCs with entries in the static table)
- Modifying wire protocol or WS topics
- Adding a `seeded` marker or system prompt hints about seeded backgrounds

## Decisions

### 1. Where seeding happens: persistence layer migration on first load

**Decision**: Add seeding logic in `talker_game_persistence.script :: load_state()`. When `saved_data.compressed_memories` is nil (brand new save), after initializing the empty memory store, iterate the static seed table and populate `memory_store_v2` backgrounds.

**Rationale**: This is where all other store migrations live (v1→v2→v3). The engine is fully initialized at `load_state` time — `alife()` is available, `story_objects.object_id_by_story_id` is populated. Seeding runs exactly once per save file, populating all ~120 unique NPCs in one pass. On subsequent loads, data is already persisted.

**Alternative considered**: Runtime intercept in Python's `_handle_background()`. Rejected because it adds complexity to the Python side (traits storage, race condition guards, mutate roundtrips) and the seed data is better authored as a static Lua table.

### 2. Static data file: AI-agent-generated Lua table

**Decision**: Create `bin/lua/domain/data/unique_backgrounds.lua` — a static table keyed by tech_name, containing `{backstory, traits, connections}` for each unique NPC. Generated by AI agent using `texts/backstory/unique.py` as source material.

**Rationale**: The agent reads the existing backstory texts, rewrites them in GM briefing style, extracts traits, and maps connections from cross-references + STALKER universe knowledge. The result is a pure data module with no engine dependencies — testable and reviewable.

### 3. Backstory rewrite style: GM briefing

**Decision**: Backstory texts are rewritten from the encyclopedic second-person style of `unique.py` into GM briefing style — atmospheric, dramatic, emphasizing personality hooks and dramatic tensions rather than factual completeness.

**Rationale**: The `unique.py` texts read like wiki articles. GM-style rewrites give the NPC things to *feel* about, not just *know* — better hooks for emergent dialogue. Same factual content, different voice.

### 4. Seed background structure: no special markers

**Decision**: The seeded background has this shape:
```lua
{
    backstory = "You've stood watch over Rookie Village...",
    traits = {"protective", "veteran", "vigilant", "pragmatic"},
    connections = {
        {name = "Sidorovich", id = "esc_m_trader",
         relationship = "Bankrolls your operations but you know he'd sell you out for the right price"},
    },
}
```

No `seeded = true` marker. These are first-class backgrounds, indistinguishable from LLM-written ones.

**Rationale**: The LLM should treat these as the character's actual background, not as something to be "enriched". No special system prompt guidance needed. The LLM can still update them via `background(action="update")` as gameplay evolves.

### 5. Tech_name → game_id resolution via story_objects

**Decision**: At `load_state` time, resolve each tech_name to a game_id using:
1. `story_objects.object_id_by_story_id[tech_name]` — O(1) lookup, works for most unique NPCs
2. Fallback: scan `alife()` objects matching `se_obj:section_name()` for MLR/addon NPCs without story_ids

Seed into `memory_store_v2` using the resolved game_id as the character key.

**Rationale**: `story_objects` is fully populated by `load_state` time. The fallback scan only runs for misses, and since this is a one-time-ever operation the cost is acceptable.

### 6. Connection id fields use tech_names

**Decision**: Each connection's `id` field stores the tech_name (e.g. `"esc_m_trader"`), not the game_id.

**Rationale**: Tech_names are stable across saves and human-readable. When Python reads them via `background(read)`, they're opaque strings the LLM can reference. The LLM can enrich connections over time.

## Risks / Trade-offs

- **[Existing saves]** → Seeding only fires on brand new saves. Players with existing saves keep their current backgrounds. Could add an MCM "re-seed unique backgrounds" option later.
- **[MLR NPC resolution]** → Some addon NPCs (e.g. `esc_main_base_trader_mlr`) may not have story_ids. The alife fallback scan handles this, but if an NPC hasn't been spawned yet in a fresh save, they'll be skipped. Acceptable — they'll get backgrounds through normal LLM interaction.
- **[Data size]** → ~120 entries with backstories of 200-500 chars each + traits + connections. Total table size ~100-150KB in Lua. Negligible memory impact.
- **[Backstory drift]** → If `unique.py` is updated, the Lua table won't auto-update. The static table is the canonical source for backgrounds; `unique.py` remains for the candidate list truncation.
